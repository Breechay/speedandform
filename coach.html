Phase 2.5 — Structural Layer (Pre-Integration Requirement)
Must exist before Strava or Garmin
Why this comes first
The current system has Intended + behavioral aftermath. That's not enough. Once Strava or Garmin enters, the device will define reality by default — unless the three-layer model already exists to contain it. Build the gate before the integration. Not the other way around.
Layer 1 of 3 — Required
Three-layer execution data model
Intended — what FORM prescribed (already exists). Perceived — what the athlete says happened (sacred, never overwritten by device). Observed — what the device recorded (optional, informs, never decides). These three must be structurally separate. FORMClosure is the atomic unit: perceived is required, observed is optional, neither overwrites the other.
Layer 2 of 3 — Required
FORMExecutionState gate
.unknown — system has no signal. .needsClosure — system has signal, meaning is missing (this is the entire product). .closed — system can proceed. Nothing downstream resolves until .closed: tomorrow logic, readiness updates, constraint states. The Close Sheet was implied before. This makes it structurally unavoidable.
Layer 3 of 3 — Required
Naming convention lock
Language in code reflects doctrine. Never use: sync, import, complete, workout. Always use: observe, close, session, record. Examples: observeRun() not syncActivity(). closeSession() not completeWorkout(). FORMObservedExecution not FORMImportedData. This prevents future drift when other developers touch the code.

System Context

What this is, who it's for, and what must not be broken. This is not philosophy — it is the operating constraint for every decision in this codebase.

FORM is not a consumer fitness app. It is the software layer for a coaching system that already exists and already works.
The methodology — Ghost Protocol, FORM mode, session structure, language — was developed and validated with real athletes before the app existed. The app is not trying to discover what works. It is expressing what already works.
Two truths that govern everything
Truth 1
The moat is the methodology, not the app.
Ghost Protocol and FORM mode are a coherent coaching philosophy expressed in software. They cannot be reverse-engineered from the UI. The app is the surface. The system is the asset.
Truth 2
This does not optimize for mass appeal.
The product does not explain itself, guide beginners step-by-step, or fight for retention. It assumes seriousness — or allows athletes to self-select in. That is the model.
Primary users
1
Coached athletes — already understand the system
These athletes trained under the methodology via PDF and site before the app existed. They have coach context. The app extends and formalizes what they already know. Do not add explanation for their benefit — they don't need it and it would feel wrong.

Self-selected athletes — discover via Ghost Protocol, no coach context
These athletes arrive without the scaffolding a coached relationship provides. The app must replace that context — through tone, structure, and precision — without becoming explanatory or noisy. Ghost Protocol's 6-week arc is doing that work. The retention system, the interruption intelligence, the graduation handoff — all of it is the coaching relationship approximated in software.
The design constraint — one sentence
When in doubt: do not add guidance, do not add explanation, do not add features. Reduce, clarify, or stay silent.
What must not be broken
Non-negotiable
Tone
Quiet, direct, never motivational. The language assumes the athlete already decided to show up. It never tries to convince them.
Restraint
Silence is correct when there is nothing to say. The system does not fill space. Empty is not a design problem to fix.
Structure
The session architecture — phases, weeks, day types — is the product. It is not a container for features. Do not restructure it to make room for new things.
Sequence
PDF → Site → App → (later: deeper system). The product matured through coaching before it became software. Do not reverse this by treating the app as the primary artifact.
The failure mode
The app becomes more helpful, more expressive, more like other running apps.
This is failure — not a version of success. Every generic UX improvement, every added explanation, every "helpfulness" feature moves the product toward Runna-lite or NRC-lite. That is a worse product for the athletes this is actually for.
The success condition
A serious athlete opens the app and feels understood. Not managed. Not impressed. Not sold to. If they feel like they're being coached, the product is working. If they feel like they're using an app, something drifted.
The stranger question — answered
"What about a stranger who downloads the app with no coaching context?"
The app does not adapt to them. It does not lower the bar, add explanations, or soften its expectations. It lets them self-select in — or self-select out. Most products fight for retention of misaligned users. This system filters for alignment instead.
The Ghost Protocol 6-week arc is what does the self-selection. A stranger who completes it has already demonstrated they're the right kind of athlete. An athlete who leaves in week 2 was not going to thrive in the system regardless. That is not a retention problem — it is the system working correctly.
The Alignment Rule
Operational — not philosophy
If a feature or change is proposed to help a user who does not already naturally align with the system —
Do not build it.
The system is allowed to lose users who require explanation, motivation, or persuasion to stay. Retention is not a goal. Alignment is the goal.
The user this rule protects against
The almost-aligned athlete — the one who "kind of gets it," "just needs a bit more explanation," "would stay if it were clearer." Optimizing for this user breaks tone, breaks restraint, and breaks self-selection. They are not the target. They are the trap.
When revenue pressure enters
The risk is not a conscious decision to chase misaligned users. It is ten small decisions that each seem reasonable — improve onboarding, reduce drop-off, add clarity — whose aggregate is drift. This rule exists for that moment. Apply it then, not just now.

The principle
Build the simplest thing that protects athlete data today. Plan the thing that scales to web. Don't build the web thing until athletes are asking for it.

Why KV store, not CloudKit (yet)
NSUbiquitousKeyValueStore is a direct drop-in for UserDefaults. CloudKit is a database.
KV store has a 1MB limit per app. A year of threshold sessions + long runs + easy weeks is well under 50KB. So the limit is irrelevant for now. The bigger reason: KV store requires zero server setup, zero auth, zero schema, and syncs automatically across all of a user's devices signed into the same iCloud account. That's what athletes actually need — not a web view. The web view is a different product decision, not a data problem to solve today.

When athletes are asking to see their data on speedandform.com. Not before. Not because it sounds like a good idea. When the demand signal is clear and repeated. That's the gate.
What CloudKit gives you
Private database per iCloud account — only the athlete sees their own data
CloudKit JS — speedandform.com can query it directly from the browser, no server required
Real-time sync across all devices including web
Record types with full query support — not just key-value pairs
Scales to thousands of users with zero backend infrastructure you maintain
What CloudKit costs you
CKRecord schema design — you have to think in records, not JSON arrays
Conflict resolution — multiple devices writing the same record simultaneously
Migration complexity — changing a record schema after data exists is delicate
Debugging is opaque — CloudKit errors are harder to diagnose than UserDefaults
Approximately 10× the implementation effort vs KV store
The migration path — additive, not destructive
1
Keep KV store as-is for the local + cross-device sync that already works. Don't remove it.
2
Add a CloudKit write alongside the KV write in save(). Same data, two destinations. LedgerEntry is already Codable — map it to CKRecord fields directly.
3
On speedandform.com, use CloudKit JS to query the athlete's private database. The athlete authorizes with their Apple ID. No FORM server involved.
4
Once CloudKit is the primary source of truth, deprecate the KV store writes. Not before.

Strava — iOS Technical Reality
How detection actually works
Strava is not a real-time sensor. Detection arrives via webhook (Strava pings your backend when an activity uploads) → your backend sends a silent push → iOS wakes app (not guaranteed). Primary detection surface must be foreground polling on app open. Background push is acceleration, not primary path. Typical lag: 5 minutes to several hours depending on device sync, upload method, and iOS background state.
Failure modes — design for these
Delayed detection — prompt arrives hours later. Use "when we last checked" not "right now".
Token expiration — never a scary error. "We lost the ability to notice your runs. Reconnect if you want."
Double activities — Garmin → Strava creates two events for one run. Deduplicate by start time ± window + duration similarity.
Activity type noise — filter: ignore under 5 min, check sport type before prompting.
Already closed manually — if Close Sheet completed before Strava event arrives, ignore the event entirely. Manual entry wins.
Deduplication hierarchy
If both Garmin and Strava fire for the same session: Garmin = source of truth for structured sessions (threshold, intervals, long run). Strava = source of truth for easy runs (low structure, duration only reliable). Same start window ± 15 min → suppress Strava if Garmin already fired.
Confidence model
Garmin structured workout → high confidence for interval segmentation. Strava easy run → high confidence for duration only. Use each source for what it's good at. Ignore what it's bad at.
