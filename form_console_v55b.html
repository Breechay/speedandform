Session Copy Standards
Apply when writing or editing any session card language
Session card language is the highest-leverage surface in the app. This is where coaching is either replaced or outsourced to the athlete. Every word must pass both tests below before being accepted.
The Fatigue Test
Read the session card as if you are 70% into the run. Breathing elevated. Cognition dropping. If any line requires thought to interpret, rewrite it. If the athlete would need to re-read it mid-session, it is not clear enough. The standard is: read once, never return to it.

This section documents the support chip system — fuel, sleep, electrolytes, shoes, recovery, cues, after/during session chips. A separate architecture from the pace drawer above. Both use the drawer UI but serve different functions. Authored in v10, carried forward as the canonical reference for all non-pace chip work.
System state
The chip drawer system is now a single architecture with one content model, one rendering path, one grammar. Every chip — fuel, sleep, recovery, electrolytes, shoes, cues, after-session, during-session — now resolves through the same pipeline.
This is not a visual change. It's structural. The system can now be extended, authored against, and audited consistently. Adding a new chip family requires: a new FORMChipKind case, a classifier check, an authored content function, and a case in the resolver switch. Four things. Consistent every time.

Infrastructure

Data persistence, sync, and the path toward speedandform.com athlete data. Three phases. Each one earns the next. Don't build what you haven't earned yet.

The principle
Build the simplest thing that protects athlete data today. Plan the thing that scales to web. Don't build the web thing until athletes are asking for it.
iCloud KV is enough for now. CloudKit is the right answer for the web. They share the same data model, so the migration is additive, not destructive. Don't skip to CloudKit — it's 10× the complexity and zero additional value until a web view exists.
Phase 1 — iCloud KV Sync
Code done · Entitlement needed
What's built
FORMLedgerStore: dual-write to UserDefaults + NSUbiquitousKeyValueStore on every save
FORMCycleStore: same dual-write pattern
Cloud-first read on load (iCloud → UserDefaults fallback)
syncFromCloud() called on scenePhase == .active
Profile note: "Training history syncs via iCloud."
One step remaining
NSUbiquitousKeyValueStore silently does nothing without the entitlement. The code exists. The entitlement does not yet.
Xcode: Target → Signing & Capabilities → + Capability → iCloud → ☑ Key-value storage. Then: build, install, log sessions, uninstall, reinstall. Data should restore. That's the test.
Why KV store, not CloudKit (yet)
NSUbiquitousKeyValueStore is a direct drop-in for UserDefaults. CloudKit is a database.
KV store has a 1MB limit per app. A year of threshold sessions + long runs + easy weeks is well under 50KB. So the limit is irrelevant for now. The bigger reason: KV store requires zero server setup, zero auth, zero schema, and syncs automatically across all of a user's devices signed into the same iCloud account. That's what athletes actually need — not a web view. The web view is a different product decision, not a data problem to solve today.
Phase 2 — CloudKit (When Web View Becomes Real)
Don't build this yet
When to build Phase 2
When athletes are asking to see their data on speedandform.com. Not before. Not because it sounds like a good idea. When the demand signal is clear and repeated. That's the gate.
What CloudKit gives you
Private database per iCloud account — only the athlete sees their own data
CloudKit JS — speedandform.com can query it directly from the browser, no server required
Real-time sync across all devices including web
Record types with full query support — not just key-value pairs
Scales to thousands of users with zero backend infrastructure you maintain
What CloudKit costs you
CKRecord schema design — you have to think in records, not JSON arrays
Conflict resolution — multiple devices writing the same record simultaneously
Migration complexity — changing a record schema after data exists is delicate
Debugging is opaque — CloudKit errors are harder to diagnose than UserDefaults
Approximately 10× the implementation effort vs KV store
The migration path — additive, not destructive
1
Keep KV store as-is for the local + cross-device sync that already works. Don't remove it.
2
Add a CloudKit write alongside the KV write in save(). Same data, two destinations. LedgerEntry is already Codable — map it to CKRecord fields directly.
3
On speedandform.com, use CloudKit JS to query the athlete's private database. The athlete authorizes with their Apple ID. No FORM server involved.
4
Once CloudKit is the primary source of truth, deprecate the KV store writes. Not before.
Phase 3 — Athlete Web View on speedandform.com
Earn this — don't plan it
Claude's synthesis — what this actually becomes
The web view is not a dashboard. It's a mirror for athletes who want to look back.
The strongest version of the speedandform.com athlete view is not a stats page — it's the same mirror card logic that exists in the Program Close Sheet, rendered on the web after a completed cycle. The athlete closes a program, the close sheet generates the identity sentence and week arc, and they can access that from the web. That's consistent with everything FORM already does: recognition, not scoring. The web view should feel like opening a finished chapter, not checking a dashboard. That framing also means you don't need to build it until the Program Close Sheet is battle-tested in the app first — the same intelligence engine feeds both.
What it should show
Completed program cycles. Identity sentence per cycle. Week arc. Field marks. Race results if any. The same mirror card — just on a larger screen.
What it should never show
Live training dashboard. Pace trends over time. Charts. VO2 estimates. Progress bars. Streaks. Anything that turns recognition into optimization.
The gate to build it
At least 5 athletes have used the Program Close Sheet and said something unprompted about it. The data story is only worth telling on the web if athletes already feel it in the app.

Long Run Intelligence

Long runs are not just duration. They are organization over time — often inside a shared containment field. When the field is missing, interpretation pressure increases. The app's job is to name that correctly.

The brief
Jose ran the Gainesville long run solo. Called it "weak mental." Missed the gang. That is not a fitness problem. The containment field was missing and he filled the silence with self-judgment. The app should correct interpretation — not add motivation.
The insight is not "people need encouragement." It is: solo changes interpretation pressure. Remove the group and you remove rhythm anchoring, emotional buffering, pacing regulation. The system gets louder and heavier. Athletes fill that silence with identity — "I suck" — instead of context — "the field was missing."
Doctrine
Core frame
The frame is not "solo is harder than group." The frame is: the absence of the expected field changes interpretation pressure, not fitness. Solo long runs are valid. The copy must never imply running alone is a lesser experience — only that it changes what the athlete hears internally. Context, not character.
What the app must never do
✕ Add motivation or encouragement
✕ Analyze pace or splits
✕ Say "solo is harder"
✕ Frame group as superior
✕ Use identity language ("mental weakness")
✕ Invent social features
✕ Guess solo context from data absence
What the app does instead
✓ Shift the before-line when context is solo
✓ Inject one containment cue during
✓ Classify outcome: contained / uneven / drifted
✓ Name the field absence explicitly in post-run copy
✓ Protect identity — context not character
✓ Default to neutral unless context is known

Strava and Garmin are not integrations. They are observers. The Close Sheet remains the center of truth. Data informs — it never decides.

The doctrine
FORM sets intent. The athlete executes. The device observes. FORM asks for meaning. Step four is non-negotiable.

The Three-Layer Model
Build this before anything else
Layer 1 — Intended
What FORM prescribed
Governs structure. Already exists. FORMSession. Type, structure, paces, intent. This layer is never touched by device data.

What the athlete says happened
Governs meaning. FORMPerceivedExecution. Effort, completion, notes, perceived segments. Required. Never overwritten by device. If perceived conflicts with observed — perceived wins.

What the device recorded

Strava — iOS Technical Reality
How detection actually works
Strava is not a real-time sensor. Detection arrives via webhook (Strava pings your backend when an activity uploads) → your backend sends a silent push → iOS wakes app (not guaranteed). Primary detection surface must be foreground polling on app open. Background push is acceleration, not primary path. Typical lag: 5 minutes to several hours depending on device sync, upload method, and iOS background state.
Failure modes — design for these
Delayed detection — prompt arrives hours later. Use "when we last checked" not "right now".
Token expiration — never a scary error. "We lost the ability to notice your runs. Reconnect if you want."
Double activities — Garmin → Strava creates two events for one run. Deduplicate by start time ± window + duration similarity.
Activity type noise — filter: ignore under 5 min, check sport type before prompting.
Already closed manually — if Close Sheet completed before Strava event arrives, ignore the event entirely. Manual entry wins.
Deduplication hierarchy
If both Garmin and Strava fire for the same session: Garmin = source of truth for structured sessions (threshold, intervals, long run). Strava = source of truth for easy runs (low structure, duration only reliable). Same start window ± 15 min → suppress Strava if Garmin already fired.
Confidence model
Garmin structured workout → high confidence for interval segmentation. Strava easy run → high confidence for duration only. Use each source for what it's good at. Ignore what it's bad at.

{ "sessions": [ { "sessionId": "S01", "weekNumber": 1, "title": "Rhythm", "understanding": "...", "do": "...", "fixItIf": "...", "doingItRightIf": "...", "takeThisWithYou": "...", "precisionNote": "...", // null if absent "altCue": "..." // null if absent } ] }
All 26 sessions in order S01–S26. weekNumber maps to GhostCycleStore's currentWeek (1–6). precisionNote and altCue are nullable — 23 sessions have precision notes, all 26 have alt cues. Swift struct: GhostSession: Codable with matching property names.
chip_content.json — schema
Schema
{ "chips": { "fuel.longRunHot": { "primary": "Eat before you start.", "whyText": "...", "holdText": "...", // null if absent "doText": "...", // null if absent "doLabel": "Do" // or "Use" for shoes }, "sleep.interrupted": { ... }, "electrolytes.hotLongRun": { ... } } }
Flat key-value where key = chipKind.stateVariant. All existing FORMChipKind cases become key prefixes. The Swift loader does a single JSONDecoder call at app launch and caches the result. Lookup is O(1) dictionary access.
form_sessions.json — schema
Schema
{ "sessions": [ { "weekIndex": 1, "weekday": "tuesday", "label": "3×10-minute threshold", "greeting": "...", "completionMessage": "...", "plan": "...", "isCurrent": true } ] }
weekIndex ranges from −4 (Phase I week 1) to 8 (taper week 2). weekday is lowercase string. isCurrent marks the plan's current target week — used by Plan tab row styling. All Phase I, main cycle, and taper sessions in one flat array, filtered by lookup.

Swift
final class GhostSessionStore { static let shared = GhostSessionStore() private(set) var sessions: [GhostSession] = [] private init() { guard let url = Bundle.main.url( forResource: "ghost_sessions", withExtension: "json"), let data = try? Data(contentsOf: url), let decoded = try? JSONDecoder().decode( SessionsFile.self, from: data) else { // Hard crash in DEBUG, silent fallback in RELEASE assertionFailure("ghost_sessions.json missing or malformed") return } sessions = decoded.sessions } func session(id: String) -> GhostSession? { sessions.first { $0.sessionId == id } } } private struct SessionsFile: Decodable { let sessions: [GhostSession] }
Load once at init. Crash in DEBUG if JSON is missing — this surfaces authoring errors immediately during development. Silent fallback in RELEASE prevents user-facing crashes from a bad bundle. The assertionFailure / silent pattern is the right call here: you control the bundle, so missing JSON is always a dev error, never a runtime edge case.
